# PROTOTYPE - Local Vercel Blob Server

Question: can we run a local service that is compatible enough with `@vercel/blob` to develop the message-attachment file workflow without asking DevOps to provision a Vercel Blob store?

Verdict: yes for the app's basic control-plane workflow:

- `put(pathname, body, { access: 'public' })`
- `put(pathname, body, { access: 'public', multipart: true })`
- `head(pathname)`
- `head(returnedLocalUrl)`
- `list({ prefix })`, including cursor pagination and `mode: 'folded'`
- `copy(fromPathname, toPathname)`
- `del(pathname | url)`
- `@vercel/blob/client.upload(...)` client-token upload
- `@vercel/blob/client.upload(..., { multipart: true })` client-token multipart upload
- `handleUpload(...)` token generation and `onUploadCompleted` callback handling
- direct private blob `GET`, `HEAD`, and `downloadUrl` reads with bearer auth
- presigned private object `GET` and `HEAD`
- presigned single-part `PUT`
- presigned multipart create, part upload, complete, and `uploadPresigned(..., { multipart: true })`
- presigned `DELETE /?pathname=...`
- `@vercel/blob/client.uploadPresigned(...)` single-part and multipart uploads
- local upload-completed callbacks for client-token and presigned upload flows
- direct fetch of the returned local URL

How callbacks are emulated: for client-token uploads, the mock decodes the
client token generated by `handleUpload`, reads its
`onUploadCompleted.callbackUrl`, signs a `blob.upload-completed` event with
`BLOB_READ_WRITE_TOKEN`, and POSTs it back to the callback route. For local
presigned uploads, callback token payloads are round-tripped through the signed
URL query and delivered to the local callback route for app-test inspection.
This proves the normal client-token and presigned callback flows without a real
Vercel Blob store.

Known gaps:

- `@vercel/blob.get(pathname, { access })` is not locally emulated. The current SDK constructs a `https://<store>.<access>.blob.vercel-storage.com/<pathname>` URL and fetches it directly, instead of using `VERCEL_BLOB_API_URL`. For local message attachments, use the URL returned from `put()` or keep downloads behind the app's own `FileService`.
- The local mock does not enforce all token constraints, quotas, content-type rules, or provider error semantics.
- Client-token upload callbacks now fail loudly when callback signing is not
  configured or the callback endpoint returns a non-2xx response.
- Pathname and list-prefix normalization rejects traversal attempts, including
  encoded `..` segments passed through SDK-style query parameters.

Run:

```sh
npm run prototype:blob
```

Leave the local server running:

```sh
npm run prototype:blob-server
```

Local env used by the prototype:

```sh
BLOB_READ_WRITE_TOKEN=vercel_blob_rw_localstore_nonce
VERCEL_BLOB_API_URL=http://localhost:9966
```

The prototype store is `.prototype-vercel-blob-store/` and can be deleted at any time.
